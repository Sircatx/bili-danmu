## 一、项目背景与目标

- **原始需求**：把 GitHub 上的 [BanqiJane/Bilibili_Danmuji](https://github.com/BanqiJane/Bilibili_Danmuji)（Java + Spring Boot 完整弹幕姬）挂到云端，**不花钱**。
- **需求收窄**：因为免费云都要绑信用卡（见第三节），用户中途改为「只要一个**能抓取某直播间弹幕**的小工具挂在云端即可」，不一定要跑那个完整 Java 程序。
- **最终目标**：云端 24/7 抓取指定直播间弹幕，弹幕日志能换电脑查看，零成本、免绑卡。

---

## 二、关键决策与放弃的方案（含原因）

| 方案 | 结论 | 原因 |
|---|---|---|
| Oracle Cloud Always Free（2 核 12G 永久） | ❌ 放弃 | 必须绑信用卡验证身份，用户不愿意 |
| Fly.io | ❌ 放弃 | ① 也要绑信用卡；② 已取消永久免费（仅 7 天试用）；③ 脚本还踩了 `fly apps status` 不存在子命令的坑 |
| Hugging Face Spaces | ⚠️ 备选 | 100% 免绑卡，但 48 小时无访问会休眠，需 UptimeRobot 定时 ping；磁盘临时、日志须外发 |
| **Koyeb + GitHub** | ✅ **采用** | 免费 nano 实例（512MB / 0.1vCPU）**免绑卡、永不休眠**；从 GitHub 仓库直接拉 Dockerfile 部署，GitHub 同时充当代码源和日志仓库 |

**最终落地形态**：用 `bilibili-api-python` 写的轻量 Python 脚本（非原 Java 程序），以 **Koyeb Worker** 常驻跑，**游客模式**连 B 站（无需登录扫码），弹幕定时 `git push` 到 GitHub 私有仓库。

---

## 三、当前方案架构

```
Koyeb Worker (nano, 免绑卡, 永不休眠)
  └─ 运行 danmu.py
       ├─ 游客模式连接 B 站直播间 WebSocket，实时收弹幕/礼物/上舰
       ├─ 弹幕写入本地 data/danmu_<房间号>.log
       ├─ 每 PUSH_INTERVAL 秒 git push → GitHub 私有日志仓库 (bili-danmu-logs)
       └─ 监听 PORT(7860) 做 HTTP 健康检查（worker 不暴露端口，起一个无害）

换电脑查看：git clone 私有日志仓库 → 打开 data/danmu_<房间号>.log
实时查看：Koyeb 服务 Logs 标签页
```

## 四、待办（部署步骤，接手人/用户执行）

> 这一步**还没做**，GitHub 仓库和 Koyeb 服务都还没建。

1. **GitHub 建两个仓库（必须分开）**
   - `bili-danmu`：放代码（`danmu.py` / `Dockerfile` / `requirements.txt`）
   - `bili-danmu-logs`：**私有**，只存弹幕日志
   - ⚠️ 不分开会导致弹幕每 5 分钟 push 触发 Koyeb 反复重新部署

2. **生成 GitHub 令牌**（classic，勾 `repo` 权限），复制保存（只显示一次）

3. **推送代码到 `bili-danmu`**
   ```powershell
   cd C:\Project\bilibili-danmu-capture
   git init -q 2>$null
   git remote add origin https://github.com/你的名/bili-danmu.git
   git add danmu.py Dockerfile requirements.txt
   git commit -m "deploy danmu capture"
   git push -u origin main
   ```

4. **Koyeb 控制台部署**
   - 连 GitHub → 选 `bili-danmu` → Builder 选 **Dockerfile** → Instance **nano** → Service type **Worker** → Scaling **Min instances = 1**
   - 环境变量：
     | 变量 | 值 | 备注 |
     |---|---|---|
     | `ROOM_ID` | `6` | 要抓的房间号 |
     | `GITHUB_TOKEN` | 令牌 | 存为 **Secret**（🔒） |
     | `GITHUB_REPO` | `你的名/bili-danmu-logs` | 日志仓库 |
     | `GITHUB_BRANCH` | `main` | |
     | `PUSH_INTERVAL` | `300` | 推送间隔秒，想少丢可改 120 |
   - 点 **Deploy**

5. **验证**：Koyeb Logs 搜 `已推送弹幕到 GitHub`；GitHub 日志仓库出现 `data/danmu_6.log` 即成功。

## 八、注意事项 / 坑

1. **Koyeb 免费实例磁盘是临时的**，容器重启/重建本地 `data/` 会丢——但弹幕已 push 到 GitHub，启动会先 `pull` 续传，不丢。
2. 若 Koyeb 在两次 push 之间重启，那一个间隔内的弹幕会丢；把 `PUSH_INTERVAL` 调小（如 120）可减少损失。
3. 若 Koyeb 提示需要绑卡才能部署，说明你地区免费档需验证，退回 **HuggingFace Spaces**（README 旧方案思路）或 Oracle（绑卡）。
4. 弹幕是私人数据，**日志仓库务必设私有**。
5. `GITHUB_TOKEN` 务必存成 Koyeb **Secret**，别用普通变量（会进日志明文）。
